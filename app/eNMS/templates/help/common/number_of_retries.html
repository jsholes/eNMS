Number of retries specifies the number of additioanl times to run the service after it completes with Failure result.

Tip: Retries can work along with the Validation section of Step 4 to create an active polling loop. For example, imagine the case where running a slow asynchronous commandX on a device, the device raises an alarm until commandX completes, and once completed, the alarm clears. If the workflow wait until commandX is complete, this can be accomplished with 2 services: a) service 1 issues commandX, b) service 2 issues a command to show alarms with Step 4 Validation set to text match the Alarm Text -and- with Negative Logic enabled -and- Number of retries set to a high number. This will cause service 2 to fail and retry showing alarms until the alarm clears, indicating that commandX is complete. Then the workflow will continue.
